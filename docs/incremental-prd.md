# 增量 PRD：「寄信」— 寄往未来的某个时刻

> 本文件仅描述相对于当前版本的变更内容。未提及的功能保持原样不变。
>
> 当前项目版本：v0.1（Demo）。本 PRD 描述的 Email 定时投递增量已实现并在 v0.1 内持续迭代。文中 `sendAt` 字段在最终实现中统一为 `openAt`，以 [README.md](../README.md) 与实际代码为准。

---

## 1. 产品变更概要

本次改造的核心是将产品从「二维码写信工具」升级为「定时信件投递服务」。信件不再只是一个可被扫码查看的链接，而是真正能在指定时间"寄出"、通过 Email 送达收信人的数字信件。

### 一句话总结

> **在保留全部现有功能的前提下，新增 Email 定时发送能力，让信件可以在未来某个时刻自动投递到收信人邮箱。**

### 新旧定位对比

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 产品定位 | 匿名二维码写信工具 | 定时信投递服务 |
| 品牌文案 | "把说不出口的话，藏进一个二维码里" | **"寄信 — 寄往未来的某个时刻"** |
| 核心交互 | 写信 → 生成二维码 → 扫码 | 写信 → 设置定时 → 自动投递到邮箱 |
| 收信方式 | 扫码 | Email 为主，扫码为辅 |
| 默认场景 | 当面送信 | 远程定时寄信 |
| 二维码角色 | 核心功能 | 附属选项（当面送礼场景） |
| 发件人身份 | 匿名（无发件人地址） | 产品域名邮箱 `letter@jixinletter.cn` |
| 定时机制 | 仅限定时解锁查看 | 定时发送 + 定时解锁 |

## 2. 用户故事（新增 / 修改的）

### 2.1 写信场景（新增）

#### P0 — 必选

- **US-001**：作为写信人，我希望在写信页面填写收信人的 Email 地址，这样信件可以发送到对方的邮箱。
- **US-002**：作为写信人，我希望在发送前选择投递时间（立即/明天/3天后/自定义时间），这样信件可以在指定时刻送达。
- **US-003**：作为写信人，我提交信件后可以看到成功确认，并获知预计送达时间。

#### P0 — 保留功能

- **US-004**：作为写信人，我仍然可以不填 Email、使用二维码方式分享信件，保留现有的当面送礼场景。
- **US-005**：作为收信人，我通过 Email 收到的链接打开后，可以看到打字机动画展开的信件内容，与现有读信体验一致。
- **US-006**：作为收信人，如果在设定时间之前点击链接，页面显示"信件尚未送达"的状态（复用现有 `locked` 机制）。

### 2.2 发件人身份场景（新增）

#### P1 — 应有

- **US-007**：作为写信人，我看到发出的 Email 来自 `letter@jixinletter.cn`，来源可信且一致。
- **US-008**：作为收信人，我收到的 Email 主题包含写信人填写的署名，发件人显示为"寄信 <letter@jixinletter.cn>"。

### 2.3 二维码降级场景（修改）

#### P0

- **US-009**：作为写信人，我仍可以生成二维码来当面展示或打印卡片，体验与当前一致。

#### P1

- **US-010**：作为用户，二维码在页面 UI 层级中不再占据中心位置，调整为"当面送礼"的附属选项。

### 2.4 读信场景（复用）

#### P0

- **US-011**：作为收信人，从 Email 链接进入读信页面后，系统自动执行打字机动画展开信件，与现有体验一致。
- **US-012**：作为收信人，30 天后信件自动销毁，访问已销毁的链接会显示"信件已消失"提示。

### 2.5 后台自动任务场景（新增）

#### P0

- **US-013**：作为系统，定时投递任务需要可靠触发，信件在设定的时间点通过 Resend API 发送 Email。
- **US-014**：作为系统，发送失败时应有重试机制和日志记录。

## 4. 信息架构变更

### 4.1 页面结构（无新增页面，变化在现有页面内）

```
写信页（index.html#write） ← 主要改动区域
├── 收信人姓名（保留）
├── **收信人 Email**（新增，P0）
│   └── 格式校验、必填提示（填 Email 时可选 QR）
├── 信件正文（保留）
├── 署名（保留）
├── **投递时间选择器**（promoted，从定时解锁升级为定时投递）
│   ├── 立即发送（立即发 Email）
│   ├── 明天（次日 9:00 发送）
│   ├── 3天后（3 天后 9:00 发送）
│   └── 自定义（日期 + 时间选择器）
├── 信纸主题（保留）
├── **提交按钮文案**：从"生成"改为"投递信件"
└── 成功提交
    ├── **Email 模式**：显示"信件将于 X 年 X 月 X 日送达"
    └── **QR 模式**：显示二维码 + 打印按钮（保留原有）

读信页（index.html#read） ← 微小改动
├── 打字机动画展开（保留）
├── **来源标识**（新增，P1）
│   └── "这封信于 X 年 X 月 X 日寄达"（来自 Email）或"当面送达"（来自 QR）
└── 锁定状态文案调整（适配新定位）
```

### 4.2 核心流程变更

```
改造前：
  写信 → 生成二维码 → 扫码 → 读信（打字机动画）

改造后：
  ┌─ 写信 → 填写 Email → 设置定时 → Email 投递 → 打开链接 → 读信
  └─ 写信 → （不填 Email）→ 生成二维码 → 扫码 → 读信（保留）
```

### 4.3 路由结构（无变化）

- `#write` — 写信页
- `#token/{id}` — 读信页
- 路由结构不变，Email 链接直接指向 `https://xxx/#token/{id}`

### 4.4 数据模型变更

在现有 `letters` 集合的文档结构中新增字段：

```javascript
// 现有字段（保留不变）
{
  _id: String,          // 短 ID
  to: String,           // 收信人名称
  body: String,         // 信件正文
  from: String,         // 署名
  theme: String,        // 信纸主题
  date: String,         // 日期
  expireAt: Date,       // 过期时间（30天后）
  createTime: Date,     // 创建时间
  openAt: Date,         // 定时解锁时间（已有，可选）

  // 新增字段
  email: String,        // 收信人 Email（可选，不填则走 QR 模式）
  sendAt: Date,         // 定时投递时间
  sent: Boolean,        // 是否已发送 Email
  sentAt: Date,         // 实际发送时间
  sendAttempts: Number, // 发送尝试次数
  sendError: String,    // 最后一次发送失败原因
}
```

### 4.5 云函数变更

| 云函数 | 变更类型 | 说明 |
|--------|----------|------|
| `saveLetter` | 修改 | 接收 email / sendAt 字段，写入数据库 |
| `readLetter` | 无修改 | 保留现有 locked / 过期逻辑 |
| `deleteLetter` | 无修改 | 保留现有逻辑 |
| **`emailLetter`** | **新增** | 查询待发送信件 → 调用 Resend API → 标记已发送 |

#### `emailLetter` 函数行为

```
1. 每分钟触发（CloudBase 定时触发器）
2. 查询 letters 集合中 sendAt <= now && sent == false 且 email 不为空的文档
3. 对每个匹配文档：
   a. 调用 Resend API 发送 Email
   b. Email 正文含信件链接 https://xxx/#token/{id}
   c. 发送成功 → 标记 sent = true, sentAt = now
   d. 发送失败 → sendAttempts++，记录 sendError
   e. 重试 3 次后仍失败 → 标记 sent = 'failed'
4. 返回处理结果统计
```
